# Технический обзор проекта "Farm Sales"

## 1. Архитектура системы

Проект представляет собой классическое клиент-серверное приложение (SPA + REST API), спроектированное с учетом принципов независимого масштабирования слоев.

*   **Frontend-слой (Presentation Layer):** Одностраничное приложение (SPA) на базе React. Отвечает за ролевой рендеринг интерфейса (UI), состояние пользователя и клиентскую валидацию. Взаимодействует с сервером по протоколу HTTP/HTTPS.
*   **Backend-слой (Application Layer):** Монолитное Java-приложение на базе Spring Boot. Реализует бизнес-логику, управление транзакциями, контроль доступа (RBAC) и интеграцию со сторонними сервисами (OSM).
*   **Слой данных (Data Layer):** Реляционная база данных PostgreSQL. Обеспечивает персистентное хранение бизнес-сущностей с поддержкой транзакционной целостности (ACID).

### 2.2. Frontend
*   **Ядро:** React 18, Vite 5

## 3. Бизнес-логика: Жизненный цикл заказа

Основа системы — это конечный автомат состояний заказа (Order State Machine):

1.  **`CREATED` (Создан):** Заказ сформирован Директором магазина. Происходит физическое резервирование товаров на складе (`Product.stockQuantity` уменьшается).
2.  **`APPROVED` (Одобрен):** Менеджер проверяет заказ и утверждает его. На этом этапе формируется финансовая и аудиторская отчетность.
3.  **`ASSIGNED` (Назначен):** Логист распределяет заказ на конкретного водителя (вручную или через алгоритм автоназначения).
4.  **`DELIVERED` (Доставлен):** Водитель отмечает выполнение заказа на месте прибытия. Заказ переходит в финальный статус, сумма заказа учитывается в выручке.

## 4. Интеллектуальная маршрутизация (Автоназначение)

Одной из самых сложных подсистем является модуль `AutoAssign` в `OrderService`. Он решает "Задачу маршрутизации транспорта" (VRP) с использованием следующих подходов:

1.  **Алгоритм кластеризации:** Группировка точек доставки вокруг "опорных" координат водителей с использованием алгоритма K-means-подобного смещения центроидов.
2.  **Min-Cost Max-Flow:** Распределение заказов между водителями как задача о потоке минимальной стоимости для равномерной нагрузки (емкость каждого водителя учитывается как ограничение потока).
3.  **Поиск пути (Haversine):** Построение цепочки доставки внутри кластера по принципу жадного поиска ближайшего соседа с использованием формулы гаверсинуса для расчета реального расстояния по сфере Земли.

---

## 5. Аудит и Realtime события

*   **Аудит (Audit Trail):** Любое изменение статуса заказа, добавление товара или регистрация пользователя логируется в таблице `audit_logs` через паттерн `ApplicationEventPublisher`.
*   **Realtime-уведомления:** Используется технология Server-Sent Events (SSE). Когда статус заказа меняется, `NotificationStreamService` мгновенно пушит событие всем активным клиентам нужной роли (например, Логист видит, что заказ одобрен, без перезагрузки страницы).

---

## 6. База данных (ER-модель)

Ключевые таблицы:
*   `users`: Хранение пользователей, ролей и хэшей паролей.
*   `products`: Справочник номенклатуры (с учетом остатков).
*   `store_addresses`: Адреса магазинов с широтой/долготой (`latitude`, `longitude`).
*   `orders`: Заголовок заказа (сумма, адреса, привязка к ролям).
*   `order_items`: Спецификация заказа (какие товары и в каком объеме).
*   `stock_movements`: Журнал движения товаров (Outbound/Inbound).
*   `order_timeline_events`: История изменений статусов каждого заказа.
