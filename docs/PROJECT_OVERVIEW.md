# Технический обзор проекта "Farm Sales"

## 1. Назначение

`Farm Sales` — многоролевая информационная система для автоматизации отдела сбыта фермерского хозяйства.

Базовый бизнес-процесс:

1. директор магазина оформляет заказ;
2. менеджер проверяет и одобряет его;
3. логист распределяет доставку по водителям;
4. водитель завершает заказ;
5. менеджер получает аналитику, аудит и Excel-отчёт.

Проект сделан как прикладная система с полным процессом, а не как набор изолированных CRUD-экранов.

## 2. Архитектура решения

Проект состоит из двух основных приложений:

- `frontend/` — SPA на React 19 + Vite 7;
- `backend/` — Spring Boot 3.4 backend на Java 21.

Хранилище зависит от контура:

- production: PostgreSQL 17;
- локальная разработка: PostgreSQL 17 через Docker Desktop и `docker-compose.yml`.

Backend организован как модульный монолит: вся бизнес-логика живёт в одном приложении, но разделена по контроллерам, сервисам и доменным областям. Для курсового проекта это проще в поддержке, тестировании и защите.

## 4. Основные сущности

- `users` — пользователи и роли;
- `store_addresses` — адреса магазинов с координатами;
- `products` — каталог товаров и логистические параметры;
- `orders` — заказы и жизненный цикл;
- `order_items` — позиции заказа;
- `order_timeline_events` — история изменений;
- `stock_movements` — движения товаров на складе;
- `audit_logs` — аудит действий;
- `realtime_notifications` — события для SSE.

## 5. Ключевая бизнес-логика

### Ролевой доступ

Доступ к API построен на Spring Security и JWT. Каждая роль видит только свой набор экранов и операций.

### Заказы и статусы

Жизненный цикл заказа:

- `CREATED`
- `APPROVED`
- `ASSIGNED`
- `DELIVERED`

Смена статуса влияет не только на UI, но и на аудит, timeline, уведомления и доступные действия следующей роли.

### Логистика и маршруты

Подсистема автоназначения обрабатывает заказы со статусом `APPROVED`, распределяет точки доставки по водителям и строит preview маршрута.

Для дорожных расстояний и геометрии используется backend-слой маршрутизации, в том числе `RoadRoutingService` и внешний OSRM-провайдер. Краткое описание алгоритма вынесено в [docs/defense/transport-task.md](./defense/transport-task.md).

### Пользователи, аудит и realtime

Менеджер может создавать директоров магазинов через отдельный пользовательский контур. Изменения ключевых сущностей публикуются в аудит и SSE-уведомления.

### Отчёты

Менеджер может выгружать `.xlsx`-отчёты по заказам. Генерация выполняется на backend.

## 6. Что важно для защиты

- проект показывает полный сквозной бизнес-процесс;
- есть управляемый reset демо-данных, который восстанавливает канонический сценарий транспортной задачи `30 точек / 30 заказов / 4498 кг`;
- роли изолированы и имеют разные рабочие области;
- присутствуют геоданные, логистика, аудит, realtime и генерация файлов.
