# Технический обзор проекта "Farm Sales"

## 1. Назначение системы

`Farm Sales` — многоролевая информационная система для автоматизации отдела сбыта фермерского хозяйства. Проект покрывает полный цикл заказа:

1. директор магазина оформляет заявку;
2. менеджер проверяет и одобряет её;
3. логист распределяет заказ по водителям;
4. водитель завершает доставку;
5. менеджер получает аналитику, аудит и Excel-отчёт.

Система ориентирована на демонстрацию прикладной автоматизации бизнес-процесса, а не на изолированные CRUD-экраны.

## 2. Архитектура решения

Проект реализован как `SPA + REST API + PostgreSQL`.

- `frontend/` — React 19 + Vite 7 приложение с ролевым интерфейсом.
- `backend/` — монолитное Spring Boot 3.4 приложение на Java 21.
- `database` — PostgreSQL 17 с миграциями Flyway.

Такой подход выбран осознанно:

- монолит проще сопровождать в рамках курсового проекта;
- роли и бизнес-правила сосредоточены в одном backend;
- REST API упрощает разделение frontend и backend при бесплатном деплое;
- PostgreSQL даёт нормальную транзакционность и реалистичную production-среду.

## 4. Основные бизнес-сущности

Ключевые таблицы и доменные объекты:

- `users` — пользователи и роли (`DIRECTOR`, `MANAGER`, `LOGISTICIAN`, `DRIVER`)
- `store_addresses` — торговые точки директоров с координатами
- `products` — каталог товаров, остатки, логистические параметры
- `orders` — заказы и их жизненный цикл
- `order_items` — позиции заказа
- `order_timeline_events` — история статусов заказа
- `stock_movements` — движения товаров на складе
- `audit_logs` — аудит действий
- `realtime_notifications` — outbox уведомлений для SSE

## 5. Жизненный цикл заказа

В системе используется конечный автомат состояний заказа:

1. `CREATED` — директор оформил заявку.
2. `APPROVED` — менеджер подтвердил заказ.
3. `ASSIGNED` — логист назначил водителя или применил автоназначение.
4. `DELIVERED` — водитель завершил доставку.

Это не просто визуальные статусы: каждая смена статуса влияет на аудит, timeline, realtime-уведомления и доступные действия следующих ролей.

## 6. Ключевая бизнес-логика

### 6.1. Ролевой доступ

Доступ к API строится на Spring Security и JWT. Каждая роль видит только свой набор экранов и может выполнять только разрешённые операции.

### 6.2. Автоназначение логиста

Подсистема автоназначения решает учебную задачу маршрутизации:

- берёт все заказы со статусом `APPROVED`;
- анализирует адреса и координаты;
- распределяет точки по водителям;
- формирует маршрутный план и позволяет подтвердить результат.

Краткое отдельное описание транспортной задачи для защиты вынесено в файл [docs/defense/transport-task.md](C:\Users\allam\OneDrive\Робочий стіл\Курсовой проект\docs\defense\transport-task.md).

### 6.3. Аудит и realtime

При логине, создании заказа, изменении статусов, создании директора и работе с товарами backend публикует события аудита. Для живого интерфейса используется SSE, поэтому менеджер и логист видят изменения без ручного обновления страницы.

### 6.4. Отчётность

Менеджер может выгружать Excel-отчёт по заказам. Отчёт строится сервером и возвращается как `.xlsx`, что удобно для демонстрации прикладной ценности проекта.

## 7. Подготовленный демонстрационный сценарий

Для защиты добавлен управляемый reset демонстрационных данных:

- backend endpoint `POST /api/demo/reset`
- PowerShell-скрипт [scripts/demo-reset.ps1](C:\Users\allam\OneDrive\Робочий стіл\Курсовой проект\scripts\demo-reset.ps1)

После reset система возвращается в каноническое состояние:

- восстановлены demo-пользователи;
- восстановлен каталог товаров;
- восстановлены demo-адреса директоров;
- заново создан транспортный сценарий с `APPROVED` заказами для логиста.

Это позволяет показывать проект преподавателю в предсказуемом состоянии, даже если до этого в системе уже создавались новые заказы, директора и отчёты.

## 8. Рекомендуемый сценарий демонстрации на защите

1. Выполнить `demo reset`.
2. Под ролью `DIRECTOR` показать профиль, адреса и оформление новой заявки.
3. Под ролью `MANAGER` показать сводку, одобрение заявки, создание директора и Excel-отчёт.
4. Под ролью `LOGISTICIAN` показать назначения и автоназначение.
5. Под ролью `DRIVER` показать маршрут и перевод заказа в `DELIVERED`.

## 9. Практическая ценность проекта

Курсовой проект демонстрирует не только базовую веб-разработку, но и:

- реальную ролевую модель;
- транзакционную бизнес-логику;
- работу с геоданными;
- генерацию файлов;
- аудит действий;
- realtime-обновления;
- деплой frontend и backend в публичную среду.

Именно это делает проект пригодным для защиты как законченной информационной системы, а не набора отдельных экранов.
