# Транспортная задача в проекте Farm Sales

## 1. Что решает алгоритм

Алгоритм нужен для роли логиста. Его задача:

- взять все заказы со статусом `APPROVED`;
- распределить их между выбранными водителями;
- учесть ограничения машины по весу и объёму;
- построить рейсы и порядок посещения точек;
- показать логисту план до записи в БД.

## 2. Где реализовано

Основные endpoint'ы:

- `POST /api/orders/auto-assign/preview` — построить план без сохранения;
- `POST /api/orders/auto-assign/route-geometry` — вернуть дорожную геометрию для карты;
- `POST /api/orders/auto-assign/approve` — применить подтверждённый план;
- `POST /api/orders/auto-assign` — сразу построить и применить план.

Ключевой класс: `backend/src/main/java/com/farm/sales/service/OrderService.java`

## 3. Какие данные подаются на вход

Алгоритм использует:

- заказы в статусе `APPROVED`;
- координаты доставки;
- выбранных логистом водителей;
- вес и объём товаров из заказа.

## 4. Основная идея решения

В проекте используется не точный математический solver, а эвристический алгоритм.

Сначала логист выбирает до трёх водителей. После этого система одновременно распределяет точки доставки между этими машинами так, чтобы суммарный дополнительный пробег был как можно меньше.

## 5. Как распределяются заказы

### Шаг 1. Выбор водителей

Логист в интерфейсе выбирает до трёх водителей, которые участвуют в расчёте.

### Шаг 2. Группировка по координатам

Если несколько заказов имеют одинаковые координаты, они объединяются в одну логистическую точку.

Для такой точки суммируются:

- вес;
- объём;
- список заказов.

### Шаг 3. Расчёт расстояний

Система строит матрицу расстояний:

- от склада до точек доставки;
- от каждой точки доставки до остальных точек.

Если точек слишком много или дорожный сервис недоступен, используется fallback по координатам.

### Шаг 4. Одновременное распределение между машинами

Во время одного рейса:

- все три машины стартуют со склада;
- для каждой машины проверяются точки, которые ещё не распределены;
- точка может быть назначена только если не превышается лимит по весу и объёму;
- на каждом шаге выбирается тот ход, который даёт минимальное добавочное расстояние для всей группы машин.

Идея здесь такая:

- заказы распределяются не по очереди “водитель 1, потом водитель 2”;
- система смотрит на всех доступных водителей сразу;
- цель — минимизировать суммарный пробег всех машин.

## 6. Как строятся повторные рейсы

Если все оставшиеся заказы уже не помещаются в текущую загрузку трёх машин:

- текущий рейс для занятых машин завершается;
- машины возвращаются на склад;
- остаток грузится заново;
- начинается следующий рейс.

Так продолжается, пока не будут распределены все заказы, которые вообще можно увезти при заданных ограничениях.

## 7. Как строится порядок точек внутри рейса

После распределения точек для конкретного рейса порядок посещения упорядочивается по принципу ближайшей следующей точки.

То есть маршрут внутри рейса тоже старается не метаться по городу, а идти компактно.

## 8. Ограничения модели

В текущей версии используются фиксированные ограничения:

- максимум `3` водителя в расчёте;
- максимум `1500 кг` на машину;
- максимум `12 м3` на машину;
- все рейсы стартуют из общего логистического хаба в Могилёве;
- возврат на склад считается только между рейсами, а после последнего рейса не добавляется;
- в демонстрационной версии используется внешний OSRM-сервис, поэтому точность дорожной геометрии зависит от его доступности.

## 9. Что получает логист

В preview логист видит:

- сколько всего одобренных заказов;
- сколько заказов вошло в план;
- сколько осталось вне плана;
- оценку суммарного километража;
- общие пояснения, почему алгоритм построил план именно так;
- отдельные рейсы по каждому водителю;
- переключение между режимом `Все водители` и маршрутом конкретного водителя;
- порядок остановок;
- причину выбора каждой точки в маршруте;
- нагрузку по весу и объёму;
- пояснение, почему конкретный рейс возвращается на склад или почему возврат не добавлен после последнего рейса;
- карту маршрутов.

После подтверждения плана:

- в заказ записывается водитель;
- статус меняется на `ASSIGNED`;
- пишутся timeline, audit и уведомления.

## 10. Как коротко объяснять преподавателю

Транспортная задача в проекте решается эвристически.

Логист выбирает до трёх водителей, после чего система одновременно распределяет между ними все одобренные заказы, учитывая вес и объём машины и стараясь минимизировать суммарный пробег. Если весь объём не помещается за один выезд, машины возвращаются на склад, берут остаток и едут в следующий рейс.
