# Транспортная задача в проекте Farm Sales

## 1. Что решает алгоритм

Алгоритм нужен для роли логиста. Его задача:

- взять все заказы со статусом `APPROVED`;
- распределить их между выбранными водителями;
- учесть ограничения машины по весу и объёму;
- построить рейсы и порядок посещения точек;
- показать логисту план до записи в БД.

## 2. Где реализовано

Основные endpoint'ы:

- `POST /api/orders/auto-assign/preview` — построить план без сохранения;
- `POST /api/orders/auto-assign/route-geometry` — вернуть дорожную геометрию для карты;
- `POST /api/orders/auto-assign/approve` — применить подтверждённый план;
- `POST /api/orders/auto-assign` — сразу построить и применить план.

Ключевой класс: `backend/src/main/java/com/farm/sales/service/OrderService.java`

## 3. Какие данные подаются на вход

Алгоритм использует:

- заказы в статусе `APPROVED`;
- координаты доставки;
- выбранных логистом водителей;
- обязательные вес и объём товаров из заказа.

## 4. Основная идея решения

В проекте используется не точный математический solver, а эвристический подход на базе Алгоритма Подметания (Sweep Algorithm) в связке с локальной оптимизацией:

- **географическая кластеризация**: точки сортируются по полярному углу относительно склада (Sweep Algorithm), чтобы каждый водитель получал компактный сектор (район) доставок;
- **жадное заполнение**: сектор "подметается", пока машина не заполнится по весу или объему;
- **локальная оптимизация (2-opt)**: порядок точек внутри выделенного сектора (рейса) выстраивается для минимизации пробега (Nearest Neighbor + 2-opt);
- **легкий post-pass rebalance**: перенос крайних точек между уже существующими рейсами, если это уменьшает общий пробег.

## 5. Как распределяются заказы

### Шаг 1. Выбор водителей

Логист в интерфейсе выбирает до трёх водителей, которые участвуют в расчёте.

### Шаг 2. Группировка по координатам

Если несколько заказов имеют одинаковые координаты, они объединяются в одну логистическую точку.

Для такой точки суммируются:

- вес;
- объём;
- список заказов.

### Шаг 3. Расчёт расстояний

Система строит матрицу расстояний:

- от склада до точек доставки;
- от каждой точки доставки до остальных точек.

Если точек слишком много или дорожный сервис недоступен, используется fallback по координатам.

### Шаг 4. Распределение секторов (Sweep Algorithm)

Алгоритм вычисляет полярный угол для каждой точки доставки относительно склада и сортирует их по кругу. Далее происходит "подметание":
- алгоритм идет по отсортированному кругу точек;
- точки назначаются текущему водителю, пока у него не закончится лимит по весу или объему;
- как только машина заполняется, алгоритм переключается на следующего водителя и продолжает "подметать" круг;
- благодаря этому водители не пересекаются маршрутами и каждый обслуживает свой собственный визуально компактный сектор (район).

## 6. Как строятся повторные рейсы

Если все оставшиеся заказы уже не помещаются в текущую загрузку трёх машин:

- текущий рейс для занятых машин завершается;
- машины возвращаются на склад;
- остаток грузится заново;
- начинается следующий рейс.

Так продолжается, пока не будут распределены все заказы, которые вообще можно увезти при заданных ограничениях.

## 7. Как улучшается уже собранный план

После первичного распределения система делает два лёгких улучшения:

- внутри каждого рейса переупорядочивает точки по принципу ближайшей следующей точки;
- затем применяет локальную `2-opt`-оптимизацию, чтобы убрать явные зигзаги;
- после этого пробует перенести отдельную точку в другой уже существующий рейс, если это уменьшает суммарный пробег и не ломает ограничения по весу и объёму.

То есть маршрут старается не только быть компактным внутри рейса, но и не оставлять явно неудачные межводительские назначения.

## 8. Ограничения модели

В текущей версии используются фиксированные ограничения:

- максимум `3` водителя в расчёте;
- максимум `1500 кг` на машину;
- максимум `12 м3` на машину;
- все рейсы стартуют из общего логистического хаба в Могилёве;
- каждый preview-рейс замыкается возвратом на склад;
- preview не гарантирует математически глобальный минимум, а строит быстрый практический план;
- в демонстрационной версии используется внешний OSRM-сервис, поэтому точность дорожной геометрии зависит от его доступности.

## 9. Что получает логист

В preview логист видит:

- сколько всего одобренных заказов;
- сколько заказов вошло в план;
- сколько осталось вне плана;
- оценку суммарного километража;
- общие пояснения, почему алгоритм построил план именно так;
- отдельные рейсы по каждому водителю;
- переключение между режимом `Все водители` и маршрутом конкретного водителя;
- дорожную геометрию только для видимого маршрута или выбранного рейса, чтобы preview открывался быстрее;
- порядок остановок;
- причину выбора каждой точки в маршруте;
- нагрузку по весу и объёму;
- пояснение, почему конкретный рейс возвращается на склад;
- карту маршрутов.

После подтверждения плана:

- в заказ записывается водитель;
- статус меняется на `ASSIGNED`;
- пишутся timeline, audit и уведомления.

## 10. Как коротко объяснять преподавателю

Транспортная задача в проекте решается эвристически на базе алгоритма подметания (Sweep Algorithm) и задачи коммивояжера.

Логист выбирает до трёх водителей, после чего система вычисляет полярный угол каждой точки доставки относительно склада. Алгоритм "подметает" заказы по кругу, отдавая каждому водителю свой компактный сектор города, учитывая ограничения по весу и объему машины. Затем внутри каждого сектора маршрут оптимизируется (Nearest Neighbor + 2-opt), чтобы минимизировать пробег. Если весь объём не помещается за один выезд, машины возвращаются на склад, берут остаток и едут в следующий рейс. Это гарантирует, что водители не будут пересекаться маршрутами и каждый будет работать в своем районе.
